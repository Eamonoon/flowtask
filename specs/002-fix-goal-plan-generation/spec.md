# 修复学习目标生成学习计划功能

**Feature Branch**: `002-fix-goal-plan-generation`

**Created**: 2026-06-23

**Status**: Draft

**Input**: 用户反馈学习目标模块存在两个问题：1) 点击"生成学习计划"按钮后只显示"创建中"无后续反馈，体验差；2) API 返回解析错误 `invalid character '`' looking for beginning of value`

## User Scenarios & Testing

### User Story 1 - 生成学习计划的即时反馈 (Priority: P1)

用户点击"生成学习计划"按钮后，系统应该提供完整的状态反馈流程，而不是仅显示"创建中"后无响应。

**Why this priority**: 这是核心用户体验问题，直接影响功能可用性。用户无法知道操作是否成功、正在生成还是失败。

**Independent Test**: 创建学习目标后，观察从点击按钮到计划生成完成的完整状态反馈流程。

**Acceptance Scenarios**:

1. **Given** 用户输入了学习目标描述，**When** 点击"生成学习计划"按钮，**Then** 按钮显示"创建中..."并禁用，同时显示加载指示器
2. **Given** 正在调用 API 创建学习目标，**When** API 返回响应，**Then** 切换到"AI 正在生成学习计划..."状态，显示流式进度
3. **Given** AI 正在生成任务，**When** 每个任务生成完成，**Then** 实时显示在页面上（显示"已生成 N 个任务..."），带流畅的动画效果
4. **Given** 学习计划生成完成，**When** 所有任务都已生成，**Then** 显示完整计划预览，提供"确认保存"和"重新生成"按钮
5. **Given** 用户查看生成的计划预览，**When** 点击"确认保存"按钮，**Then** 任务保存到数据库，显示成功状态和任务总数
6. **Given** 用户查看生成的计划预览，**When** 点击"重新生成"按钮，**Then** 清除当前结果，重新开始生成流程
7. **Given** API 调用失败，**When** 返回错误，**Then** 显示友好的错误信息和重试按钮，而不是静默失败

---

### User Story 2 - 修复 AI 响应解析错误 (Priority: P1)

系统应该能够正确解析 AI 返回的学习计划，而不是因为格式问题（如 markdown 代码块）导致解析失败。

**Why this priority**: 这是功能完全不可用的 bug，用户无法生成任何学习计划。

**Independent Test**: 使用标准的学习目标描述（如"我想两个月学会 RAG"）调用 API，验证能成功生成计划。

**Acceptance Scenarios**:

1. **Given** 用户提交学习目标，**When** AI 返回带有 markdown 格式的响应（如 ```json 代码块），**Then** 系统正确解析并提取 JSON 内容
2. **Given** AI 返回的响应包含额外文本（如解释性文字），**When** 解析响应，**Then** 只提取有效的 JSON 数据，忽略其他内容
3. **Given** AI 返回的 JSON 格式不完整，**When** 解析失败，**Then** 提供清晰的错误信息，说明具体解析问题

---

### User Story 3 - 网络中断恢复 (Priority: P2)

用户在生成学习计划过程中离开页面（关闭浏览器、切换标签页），返回时系统应该保留已生成内容并提供继续/重新生成选项。

**Why this priority**: 网络中断是常见场景，保留生成进度能避免用户重复等待，提升整体体验。但核心生成功能（US1/US2）应优先保证可用性。

**Independent Test**: 在生成过程中刷新浏览器页面，验证系统显示已生成的部分任务并提供继续/重新生成选项

**Acceptance Scenarios**:

1. **Given** 用户正在生成学习计划（已生成 3 个任务），**When** 刷新浏览器页面，**Then** 页面显示已生成的 3 个任务，并提供"继续生成"和"重新生成"按钮
2. **Given** 用户看到已生成的部分任务，**When** 点击"继续生成"按钮，**Then** 系统重新连接到现有 session，继续生成剩余任务
3. **Given** 用户看到已生成的部分任务，**When** 点击"重新生成"按钮，**Then** 清除已生成内容，重新开始生成流程
4. **Given** 用户的生成 session 已过期（超过 24 小时），**When** 返回页面，**Then** 系统提示"生成会话已过期"，提供"重新生成"选项

---

### Edge Cases

- AI 返回空响应时如何处理？
- AI 返回的 JSON 包含特殊字符（转义字符、换行符）时如何处理？
- 网络超时或中断时，显示已生成的部分任务，并提供"继续生成"按钮让用户选择是否继续
- 用户在生成过程中离开页面后返回，保留已生成内容并提供继续/重新生成选项
- AI 返回的任务结构不符合预期格式时如何处理？

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 在用户点击"生成学习计划"后立即提供视觉反馈（按钮状态、加载指示器）
- **FR-002**: 系统 MUST 支持流式显示 AI 生成的学习任务，每个任务生成后实时展示
- **FR-003**: 系统 MUST 在生成过程中显示当前进度（如"已生成 3 个任务..."，不预估总数）
- **FR-004**: 系统 MUST 能够解析 AI 返回的多种格式（纯 JSON、markdown 代码块、带解释文本的 JSON）
- **FR-005**: 系统 MUST 在解析失败时提供友好的错误信息，包含可能的原因和重试建议
- **FR-006**: 系统 MUST 在生成失败时允许用户重试，而不是需要重新输入学习目标
- **FR-007**: 系统 MUST 在生成完成后显示完整计划预览，等待用户确认
- **FR-008**: 系统 MUST 在用户确认后才将任务保存到数据库
- **FR-009**: 系统 MUST 允许用户在预览阶段选择重新生成计划
- **FR-010**: 系统 MUST 在网络中断后保留已生成内容（24 小时有效期），提供继续生成选项

### Key Entities

- **LearningGoal**: 学习目标，包含描述、目标时长、状态（active/completed/paused）
- **Task**: 学习任务，包含标题、描述、父任务、排序顺序、预估时长
- **GenerationState**: 生成状态，包含 phase（idle/connecting/streaming/done/error）、进度、错误信息

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户从点击"生成学习计划"到看到第一个任务的时间 < 3 秒
- **SC-002**: 生成过程中，用户能实时看到任务逐步出现，无需等待全部完成
- **SC-003**: API 解析错误率 < 1%（基于 AI 返回的各种格式）
- **SC-004**: 生成失败时，100% 的情况显示明确的错误信息和重试选项
- **SC-005**: 用户满意度：生成流程的反馈清晰度评分 > 4/5（通过上线后用户反馈问卷测量，问卷在首次使用后弹出）

## Clarifications

### Session 2026-06-23

- Q: 用户在计划生成过程中离开页面后返回，系统应该如何处理？ → A: 显示已生成部分 + 提供"继续生成"按钮
- Q: AI 生成的学习计划显示后，用户能否立即编辑或调整任务顺序？ → A: 确认后再保存：生成后显示预览，用户确认后才保存到数据库
- Q: 由于 AI 生成任务时无法预知总数，进度应该如何显示？ → A: 只显示已生成数量：如"已生成 3 个任务..."

## Assumptions

- AI 服务（OpenAI 或兼容 API）已正确配置并可用
- AI 返回的响应是有效的 JSON 或可解析的 JSON 变体（如 markdown 代码块）
- 用户的网络连接稳定，能够完成流式传输
- 学习目标的描述长度在合理范围内（10-500 字符）
- AI 生成的任务数量通常在 5-20 个之间

## Dependencies

- 后端 API 服务正常运行
- AI 服务已配置并可用
- 数据库和缓存服务可用
- 前端框架和依赖已安装
